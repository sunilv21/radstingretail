## Findings

### P2 - Lint gate is broken

Files:
- `package.json`
- `eslint.config.mjs`

Evidence:
- `package.json` uses `"lint": "next lint"`.
- Running `npm run lint` fails with `Invalid project directory provided, no such directory: ...\lint`.
- `eslint.config.mjs` already notes that Next.js 16 removed `next lint`.
- `eslint` is not installed in `node_modules`.

Impact:
- CI/local QA cannot run lint.
- Next-specific UI/accessibility checks are absent.

Expected fix:
- Install ESLint packages and change the script to run `eslint .`, ideally with `eslint-config-next`.

### P2 - `LedgerEngine.verifyBalance()` crashes in ESM

File:
- `server/engines/ledger.engine.js`

Evidence:
- Direct call: `LedgerEngine.verifyBalance('000000000000000000000000')`
- Actual: `ReferenceError: require is not defined in ES module scope`
- Cause: the function calls `require('mongoose')` inside an ESM project.

Impact:
- Any future route/script using this engine helper will fail.
- The public `/api/reports/ledger-balance` route is currently safe because it implements its own aggregation.

Expected fix:
- Import `mongoose` at module top and use `new mongoose.Types.ObjectId(storeId)` or the existing route-style helper.

### P3 - Canonical test plan has stale platform endpoint

File:
- `docs/test-cases.md`

Evidence:
- TC-API-500 expects `GET /platform/tenants`.
- Backend implements `GET /platform/organizations`.
- Admin frontend also calls `/platform/organizations`.

Impact:
- Manual QA will report a false failure for the admin tenant list.

Expected fix:
- Update TC-API-500 to `/platform/organizations`, or add a backward-compatible `/platform/tenants` alias.

## Blocked Coverage

The following were not executed end-to-end because they mutate the configured Atlas database or require external credentials/devices:

- Creating sales, returns, voids, purchase orders, GRNs, payments, stock adjustments, transfers, users, branches, and subscriptions.
- Button-by-button browser interaction and visual regression screenshots.
- WhatsApp, PhonePe, Razorpay, GSP/e-invoice live flows.
- Security tests that attempt mutation, rate-limit exhaustion, or cross-tenant writes.
- Performance tests such as 1000-item cart, simultaneous cashiers, and kill/rollback tests.

Recommendation: run the full manual checklist in `docs/test-cases.md` against a disposable QA database seeded with known fixtures, then allow write tests.
